Moving from a "send-everything" approach to a more surgical architecture will solve both your accuracy issues and your token costs. Even powerful models like Claude can struggle with "needle-in-a-haystack" problems when you dump a massive JSON/CSV into the context window, and as you've noticed, it's expensive.To improve accuracy while lowering tokens, we should shift from CAG (Context-Augmented Generation) to an Agentic Tool-Use or NL2Code (Natural Language to Code) architecture.1. The Strategy: "Don't Send Data, Send Metadata"Instead of sending the actual rows and values, send only the schema (column names, data types, and a few sample values). Then, ask the LLM to generate a specific script or query to fetch exactly what it needs.The New WorkflowSchema Extraction: Your Office.js frontend extracts the headers and a 1-row sample.Intent Classification: The LLM receives the prompt + Schema. It decides: "Do I need to calculate something, or just find a value?"Code Generation: The LLM returns a small snippet of JavaScript (Office.js) or a specific JSON command (e.g., {"action": "find", "column": "X", "row_criteria": "Y"}).Local Execution: Your Add-in executes that code locally in Excel.Small Context Final Answer: The Add-in sends only the result back to the LLM to format the final natural language response.2. Technical Implementation: The "Router" DesignTo balance cost and performance, you can use a "Hybrid" model approach in your NestJS backend:Phase A: The Cheap Router (Local Llama 3.1)Use your local Llama model specifically for Task Routing. Llama 3.1 is quite good at following a strict JSON schema if you use Grammar Constrained Output (Ollama supports this).Input: User Question + Table Headers.Output: A JSON object: { "task_type": "lookup" | "summary" | "format", "target_range": "A1:C50" }.Phase B: The Precision Strike (Claude / GPT-4o-mini)Once you know exactly which 5-10 rows are needed, send only those rows to Claude.Token Savings: You go from sending 1,000 rows (massive tokens) to sending 5 rows (minimal tokens).Accuracy: Small, relevant context leads to much higher accuracy.3. Optimizing the "Context" (Token Reduction)If you must send data, don't use JSON. JSON is extremely "token-heavy" because of the repeated keys and curly braces.Switch to Markdown Tables or CSV:JSON (Heavy): [{"Name": "John", "Age": 30}, {"Name": "Jane", "Age": 25}]CSV (Light): Name,Age\nJohn,30\nJane,25Coordinate Mapping: Instead of the model guessing where data is, prepend every row with its index: Row 2: John, 30. This helps the model cite its sources accurately.4.
